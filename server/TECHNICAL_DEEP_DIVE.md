# 🛡️ FlashSale Sentinel: Technical Deep Dive

## 1. The Ingestion Layer (Gatekeeper)

**Files:** `server.js`, `orderRoutes.js`, `orderController.js`

When a user clicks "Buy," the system prioritizes **Response Time (Latency)** over **Consistency**.

- **Zero Database I/O:** The controller never talks to MongoDB. It only talks to Redis. This keeps the request-response cycle under 50ms.
- **HTTP 202 Accepted:** We return a 202 instead of a 200. This signifies that the request is valid and has been queued, but the final result (success/fail) is pending.
- **Idempotency Token:** The `jobId` generated by the controller is the "Source of Truth" for tracking that specific request through the entire pipeline.

## 2. The Transport Layer (Broker)

**Files:** `config/redis.js`, `queues/orderQueue.js`

- **Redis/BullMQ:** Acting as a buffer, Redis prevents the database from being overwhelmed. If 10,000 requests arrive in 1 second, Redis stores them safely.
- **Persistence:** Because BullMQ uses Redis's RPOPLPUSH (or Lua scripts), no job is lost even if a worker crashes mid-process.

## 3. The Engine (Background Worker)

**Files:** `workers/orderWorker.js`

This is where the complex business logic and "Safety" checks occur.

- **ACID Transactions:** We use `mongoose.startSession()`. If the system decrements stock but fails to save the order record, the stock decrement is rolled back.
- **Atomic Decrement:** Instead of `product.stock = product.stock - 1`, we use `{ $inc: { stock: -1 } }` with a filter `{ stock: { $gt: 0 } }`. This ensures that even with 10 workers running in parallel, we never sell more items than we have (Overselling Protection).
- **Idempotency Guard:** The worker checks the `Order` collection for the `jobId` before doing anything. This prevents duplicate processing if the same job is delivered twice by the queue.

## 4. The Persistence Layer (Source of Truth)

**Files:** `config/db.js`, `models/`

- **Connection Pooling:** `maxPoolSize: 100` allows the workers to perform many parallel writes without waiting for a connection to open.
- **Indexing:** The `Order` model uses a unique index on `jobId`. This is the final database-level shield against duplicate orders.

## 5. The Feedback Loop (Real-time)

**Files:** `server.js`, `socket.js`

- **Socket.io Rooms:** When a user connects, the server places them in a "Room" based on their `userId`.
- **Queue Events:** The main server instance listens to `QueueEvents` from Redis. When a worker finishes a job, the server "hears" it and emits the result to that specific user's room.

| Component           | Metric    | Optimization Technique            |
| :------------------ | :-------- | :-------------------------------- |
| **Ingestion**       | < 50ms    | Redis `O(1)` insertion, No DB I/O |
| **Stock Safety**    | 100%      | MongoDB Atomic Operators (`$inc`) |
| **Data Integrity**  | 100%      | MongoDB Sessions (Transactions)   |
| **User Experience** | Real-time | Socket.io targeted rooms          |

## 🛑 Critical Configuration Notes

1. **Redis `maxRetriesPerRequest`**: Must be `null` for BullMQ.
2. **MongoDB `minPoolSize`**: Set to `10` to avoid "Cold Start" connection latency at the exact second a sale begins.
3. **Graceful Shutdown**: The server must close Redis and MongoDB connections in order to ensure no "Ghost Jobs" remain in memory or "Hanging Connections" in the DB.
